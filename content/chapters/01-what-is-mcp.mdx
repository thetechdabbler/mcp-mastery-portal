## The setup

You already know APIs. MCP is a **host ↔ client ↔ server** contract for attaching **tools**, **resources**, and **prompts** to models—without every IDE vendor reinventing the same brittle glue.

## Picture this

<Diagram id="mxn-before-after" caption="Before MCP: hosts re-integrate every backend. After MCP: each domain ships one server." alt="Flowchart comparing many host-backend arrows vs host through MCP client to multiple servers" />

## Mental model

Treat **Model Context Protocol** like a typed plugin surface: the model never “just calls HTTP”; the **host** mediates, the **client** speaks JSON-RPC, the **server** exposes capabilities with schemas.

## Walkthrough

### MCP vs alternatives (tradeoffs, not marketing)

| Approach | Upside | Downside |
| --- | --- | --- |
| Vendor function calling | polished DX | lock-in, bespoke discovery |
| Ad-hoc REST | maximum flexibility | no standard capability listing |
| MCP | interop + discovery | you must own security + governance |

### Comparison you actually asked for

OpenAI-style function calling is great—inside one vendor ecosystem. MCP’s bet is **multi-host, multi-server, multi-vendor** interoperability with a shared mental model.

## Hall of Shame

<HallOfShame tag="architecture" title="Integration tourism">

```txt
"We'll expose 40 REST endpoints and let the model guess URLs."
```

**Fix:** expose **tools** with explicit schemas, **resources** for reads, **prompts** for repeatable templates—then enforce auth at the boundary.

</HallOfShame>

## Why this matters in production

Interop is a cost center until it isn't. MCP moves integration work from **every host** to **one MCP server per domain**—but only if you treat schemas, auth, auditing, and supply-chain risk as first-class.
