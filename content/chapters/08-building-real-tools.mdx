## The setup

“Real tools” means **schemas**, **timeouts**, **cancellation**, and **structured errors**—not vibes.

## Picture this

<Diagram id="tool-cancel-progress" caption="Long tool: client may cancel; server can emit progress notifications." alt="Sequence diagram tool call cancel progress" />

## Mental model

Treat tools like RPC endpoints exposed to an **adaptive adversary** (the model).

## Walkthrough

- Keep handlers thin; push domain logic into testable modules.
- Prefer explicit enums over `string` for modes.
- Return errors as **machine-readable** MCP errors when possible.

## Hall of Shame

<HallOfShame tag="validation" title="cmd string strikes again">

```ts
inputSchema: { cmd: z.string() }
```

**Fix:** enumerate operations; never expose raw shells to models.

</HallOfShame>

## Why this matters in production

Every missing validation is a **silent feature** for attackers—internal or external.

## Mini challenge

Pick one tool and delete half its parameters—what breaks? That’s your real API surface.

## Reflection

Which tool in your design is actually three tools wearing a trench coat?

## You can now brag that…

You can defend a tool schema in code review without saying “the model won’t do that.”
