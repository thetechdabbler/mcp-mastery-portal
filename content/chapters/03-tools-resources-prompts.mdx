## The setup

Primitives are boring until they’re wrong—then they’re **incidents**.

## Picture this

<Diagram id="trp-decision-tree" caption="Decision tree: side effects? choose tool. stable read URI? resource. repeatable instruction? prompt." alt="Decision tree for choosing tool, resource, or prompt" />

## Mental model

- **Tools**: imperative verbs with validated args—assume hostile inputs.
- **Resources**: nouns you can **read**—great for docs, configs, code slices.
- **Prompts**: packaged reasoning patterns—parameterized, versioned, reviewed.

## Walkthrough

If you expose a giant “doAnything” tool, you didn’t discover AGI—you discovered **incident response**.

## Hall of Shame

<HallOfShame tag="validation" title="Tool as filesystem">

```ts
tool("read_file", { path: z.string() }, async ({ path }) => fs.readFile(path));
```

**Fix:** resources with roots + normalization, or tools with allowlists + realpath checks.

</HallOfShame>

## Why this matters in production

Primitives are your **authorization story**. Resources are easier to cache and audit; tools should be few and tightly scoped.

## Mini challenge

Rewrite one “read tool” in your design as a **resource** with a URI template—what breaks?

## Reflection

Where are you using a tool because MDX examples did, not because the operation is imperative?

## You can now brag that…

You can defend a TRP choice in a design review without hand-waving “the LLM will figure it out.”
