## The setup

If you blur **host**, **client**, and **server**, you’ll design “helpful” tools that bypass every safety rail the host tried to install.

## Picture this

<Diagram id="host-client-server-topology" caption="Host embeds model + UI; client multiplexes JSON-RPC; servers expose capabilities." alt="Diagram with host containing client connected to two MCP servers" />

## Mental model

**Host** owns consent and policy. **Client** translates host intent into JSON-RPC. **Server** answers with tools/resources/prompts—nothing more romantic than that.

## Walkthrough

### Responsibilities

- **Host**: launches servers (stdio) or connects HTTP, shows UI, applies org policy.
- **Client**: capability negotiation, subscriptions, tool calls, error surfacing.
- **Server**: business logic adapter + schema validation + least privilege.

## Hall of Shame

<HallOfShame tag="architecture" title="Host == Server">

```txt
"The LLM is basically the MCP host, right?"
```

**Fix:** the model proposes; the **host** decides; the **client** executes protocol; the **server** performs side effects.

</HallOfShame>

## Why this matters in production

Policy engines attach to **hosts**. If you pretend the server is sovereign, you’ll ship “helpful” remote admin tools and wonder why Security shows up with PDFs.
